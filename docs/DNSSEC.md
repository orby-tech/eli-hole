# DNSSEC в EliHole

Подробное описание реализации DNSSEC: что делаем, зачем, как устроено, что готово и что
впереди. Документ ведётся по ходу разработки (фича большая, идёт несколькими циклами).

---

## 1. Что такое DNSSEC и зачем

DNSSEC (DNS Security Extensions) — это криптографические подписи на DNS-данных. Они
гарантируют, что ответ **не подменили** по дороге (отравление кэша, MITM, спуфинг). Каждая
подписанная зона подписывает свои записи; резолвер проверяет подпись и цепочку доверия от
корня (`.`) до запрошенного имени.

Результат валидации одного ответа:

| Статус | Смысл |
|---|---|
| `:secure` | Подпись валидна, цепочка доверия от корня сошлась. Ответу можно доверять. |
| `:insecure` | Зона не подписана (нет DS-делегации). DNSSEC не применяется — обычный DNS. |
| `:bogus` | Подпись есть, но **не сошлась** (подделка / сломанная подпись / истёкшая). Ответ отдавать нельзя → SERVFAIL. |

### Выбранный режим — полная валидация (не proxy)

Было два варианта:

- **Proxy/aware** — EliHole доверяет вышестоящему резолверу (8.8.8.8 и т.п.), который сам
  валидирует, и просто читает флаг `AD` из его ответа.
- **Полный валидирующий резолвер** — EliHole **сам** проверяет криптоподписи и строит цепочку
  доверия от корня. Не доверяет никому, проверяет математикой.

Выбран **полный валидирующий резолвер**. EliHole сам считает RRSIG/DS/DNSKEY, якорится на
корневой trust anchor ICANN. На `:secure` ставит бит `AD` клиенту; на `:bogus` отдаёт SERVFAIL;
`:insecure` пропускает как обычный DNS.

## 2. Как работает цепочка доверия (chain of trust)

```
                 [хардкод: корневой trust anchor — DS корневого KSK]
                                     │ (доверяем по DS-дайджесту)
        ┌────────────────────────────▼────────────────────────────┐
   .    │ DNSKEY корня (KSK+ZSK), RRSIG(DNSKEY) подписан KSK         │
        └────────────────────────────┬────────────────────────────┘
                                      │ DS(com) в корне, подписан ZSK корня
        ┌────────────────────────────▼────────────────────────────┐
  com   │ DNSKEY .com, его KSK хешируется в DS из корня → совпало    │
        └────────────────────────────┬────────────────────────────┘
                                      │ DS(example.com) в .com, подписан ZSK .com
        ┌────────────────────────────▼────────────────────────────┐
example │ DNSKEY example.com; A-запись подписана RRSIG (ZSK)        │
 .com   └───────────────────────────────────────────────────────────┘
```

Два типа ключей в зоне:
- **KSK** (Key Signing Key, SEP-бит) — подписывает только DNSKEY RRset. На него указывает DS из
  родителя.
- **ZSK** (Zone Signing Key) — подписывает реальные записи (A, MX, …).

Звенья, которые умеет проверять движок:
- **`verify_rrsig`** — подпись RRSIG над RRset валидна для данного DNSKEY.
- **`verify_ds`** — DS из родителя = `hash(owner || DNSKEY_RDATA)` дочернего KSK.

Из этих двух примитивов в Loop 2 собирается полный обход корень → имя.

## 3. Архитектура (namespace `EliHole.DNSSEC`)

| Модуль | Файл | Ответственность | Статус |
|---|---|---|---|
| `Wire` | `lib/eli_hole/dnssec/wire.ex` | Парсер DNS-сообщений: RR с **сырым RDATA**, декомпрессия имён, биты AD/CD, сборка запроса с EDNS0 DO-битом. | ✅ Loop 1 |
| `RR` | `lib/eli_hole/dnssec/rr.ex` | Парсеры RDATA: DNSKEY/RRSIG/DS/NSEC/NSEC3; key tag (RFC 4034 App B); DNSKEY → ключ для `:crypto`. | ✅ Loop 1 |
| `Canonical` | `lib/eli_hole/dnssec/canonical.ex` | Каноническая форма RR, сортировка RRset, сборка signing input для RRSIG. | ✅ Loop 1 |
| `Crypto` | `lib/eli_hole/dnssec/crypto.ex` | `verify_rrsig` (RSA/ECDSA/EdDSA), `verify_ds`. Обёртка над `:crypto`. | ✅ Loop 1 |
| `TrustAnchor` | `lib/eli_hole/dnssec/trust_anchor.ex` | Хардкод корневых DS (KSK-2017 + KSK-2024). | ✅ Loop 1 |
| `Client` | `lib/eli_hole/dnssec/client.ex` | Живые запросы DNSKEY/DS к upstream с DO-битом (UDP + TCP-fallback на truncation) + ETS-кэш. GenServer в supervision tree. | ✅ Loop 2 |
| `Validator` | `lib/eli_hole/dnssec/validator.ex` | Обход цепочки корень→signer → `{:secure, signer} \| {:insecure, reason} \| {:bogus, reason}`. Fetcher и часы инъектируемы (offline-тесты). | ✅ Loop 2 |

### Почему свой wire-парсер, а не `:inet_dns`

Эмпирически проверено: `:inet_dns.decode` разбирает **обвязку** записи (owner/type/class/ttl),
но RDATA для DNSSEC-типов отдаёт **сырым блобом**, а биты `AD/CD/DO` **вообще не показывает**.
Для валидации нужен контроль над сырыми байтами RDATA (иначе не построить каноническую форму
для проверки подписи) и доступ к флагам. Поэтому отдельный парсер.

## 4. Что движок умеет после Loop 1

Чистый криптодвижок без интеграции в резолвер. По сырым байтам:
- распарсить DNS-ответ, достать DNSKEY/RRSIG/DS/RRset;
- посчитать key tag (совпадает с публичными 20326/38696 корня);
- проверить подпись RRSIG над RRset для RSA (alg 8/10), ECDSA (13/14), Ed25519/Ed448 (15/16);
- проверить связь DS → DNSKEY (включая корневой anchor).

Покрыто тестами на реальных захваченных ответах (cloudflare.com — ECDSA, nlnetlabs.nl и корень
— RSA) + проверки «мутировал байт → подпись не сходится».

## 5. Тонкости, которые легко сломать

Эти места — главные источники «молчаливых» провалов валидации (подпись верная, а код говорит
bogus). Все учтены:

1. **OriginalTTL из RRSIG**, а не TTL из пакета, идёт в signing input (RFC 4034 §3.1.8.1).
2. **Сортировка RRset** по канонической RDATA как по беззнаковым октетам: «отсутствие октета
   сортируется раньше нулевого октета» (§6.3) — это ровно порядок бинарей в Erlang.
3. **Даунскейс имён в RDATA** по списку RFC 6840 §5.1: имя в RRSIG (signer) — даунскейзим, имя в
   NSEC (next-name) — **НЕ** даунскейзим. Старый список §6.2 устарел.
4. **ECDSA-подпись** в DNSSEC — сырой `r‖s` фиксированной ширины (RFC 6605), а
   `:crypto.verify(:ecdsa, …)` хочет **DER** `SEQUENCE{r,s}` → оборачиваем вручную (с ведущим
   `0x00`, если старший бит установлен). Точку публичного ключа префиксуем `0x04`.
5. **RSA-ключ** (RFC 3110): длина экспоненты — 1 октет если ≤255, иначе `0x00` + 2 октета.
6. **EdDSA** — `:crypto.verify(:eddsa, :none, data, sig, [pk, :ed25519])`, хеш `:none`.

## 6. Корневые trust anchors (хардкод)

В 2026 идёт **KSK rollover** — в `root-anchors.xml` опубликованы **ДВА** валидных якоря, оба
зашиты в `TrustAnchor`:

| Имя | Key tag | Алгоритм | Digest type | DS digest (SHA-256) |
|---|---|---|---|---|
| KSK-2017 | 20326 | 8 (RSASHA256) | 2 | `E06D44B8…C7F8EC8D` |
| KSK-2024 | 38696 | 8 (RSASHA256) | 2 | `683D2D0A…C0FB2B16` |

Якорить только 20326 нельзя — сломается в окне окт-2026…янв-2027, когда подписывать начнёт
KSK-2024. KSK-2017 будет revoked 2027-01-11 — тогда обновить.
Источник: <https://data.iana.org/root-anchors/root-anchors.xml>.

## 7. Тестовые фикстуры

`test/support/dnssec_fixtures.ex` — реальные сырые DNS-ответы (base64), захваченные с DO-битом
через 8.8.8.8. Тесты офлайн-детерминированы.

Важно: **криптотесты не зависят от времени** — математика подписи валидна вечно (проверка
inception/expiration делается в `Validator`, Loop 2, а не в `Crypto`). RRset в фикстуре зафиксирован
вместе со своей RRSIG, поэтому остаётся согласованным.

Перезахват (если зона сменит ключи): захват делается тем же приёмом, что в
`/tmp/genfix.exs` — запрос с EDNS0 OPT (DO=1), сырой ответ в base64. (В Loop 2 вынесем в
`priv/dnssec/recapture.exs`.)

## 8. Дорожная карта (циклы)

- **Loop 1 — криптодвижок** ✅ *(текущий)*. `Wire`, `RR`, `Canonical`, `Crypto`, `TrustAnchor` +
  юнит-тесты на RFC-векторах и живых захватах. Без интеграции в резолвер.
- **Loop 2 — цепочка доверия** ✅. `Client` (живой fetch DNSKEY/DS + кэш, UDP+TCP) + `Validator`
  (обход root→signer, secure-путь; insecure = нет DS — best-effort, без NSEC-доказательства отсутствия,
  это Loop 4). Offline-тесты на захваченных пакетах (полная цепочка root→com→cloudflare.com, +bogus
  /expired/broken-delegation/insecure). Живая проверка — `make dnssec.demo`.

  **Важная находка про zone cuts:** Cloudflare делает **каждое имя отдельной зоной** — у
  `www.cloudflare.com` свой DNSKEY и свой DS в `cloudflare.com` (signer RRSIG = сам FQDN). Поэтому
  валидатор обходит зоны по суффиксам signer-имени и не предполагает, что зона = «домен второго
  уровня». Также: при KSK-rollover в DNSKEY-наборе несколько DS-заякоренных KSK — подписанта DNSKEY
  RRset берём **из самого RRSIG** (RFC 4035 §5.2), а не первый совпавший с DS (иначе ловим KSK-2024,
  который опубликован, но пока не подписывает).
- **Loop 3 — интеграция в резолвер + UI** ✅. `Resolver.dnssec_status/3` классифицирует каждый
  resolved-запрос (`:secure`/`:insecure`/`:bogus`), `Server` кладёт вердикт в `QueryLog` **вне
  критического пути** (ответ клиенту уходит до валидации), `QueryLogLive` рисует бейдж в колонке
  DNSSEC. Клиентский запрос обычно без DO-бита, поэтому статус считается отдельным DO-запросом через
  `Client` (кэш) + `Validator`. Любая ошибка валидации → `nil`, DNS не ломается.
  - *Пока НЕ сделано (намеренно):* enforcement (SERVFAIL на `:bogus`) и бит `AD` клиенту — сейчас
    только классификация и отображение, чтобы не рисковать резолвингом при багах валидации.
    Тумблер вкл/выкл — тоже отдельно.
- **Loop 4 — enforcement + harden** ✅ (частично). `DNSSEC.Config` — тумблер `enforce` (ETS+DB+PubSub,
  default OFF). Когда ON, `Server` валидирует **до** ответа клиенту: `:bogus` → SERVFAIL (битый ответ
  не отдаём), `:secure` → выставляем бит `AD`; `:insecure`/нет данных → как есть. Тумблер на
  `/admin/settings`. Также: перебор всех RRSIG (мультиподпись, фикс из ревью). Проверено живьём —
  cloudflare.com (secure) приходит с AD=1, google.com (insecure) обычным ответом.
- **Loop 5 — denial-of-existence (NSEC3)** ✅. Модуль `DNSSEC.Denial`: NSEC3-хеш (итеративный SHA-1
  + соль, RFC 5155 §5), base32hex, match/cover, opt-out. Пустой DS теперь **требует подписанного
  доказательства** отсутствия (`prove_ds_absence`): NSEC3 match (NS есть, DS/SOA нет) или opt-out cover
  → `:insecure` (честно неподписанная делегация); нет валидного доказательства → `:bogus` —
  **stripped-DS downgrade пойман**. NSEC (без хеша) — аналогично по owner. Хеш проверен против
  тест-векторов RFC 5155 Appendix A; downgrade — на живой фикстуре microsoft.com (.com NSEC3 opt-out).
- **Loop 6 — полировка** ✅. Лимит итераций NSEC3 (RFC 9276): `> 100` → `:insecure` без хеширования
  (защита от CPU-DoS на раздутом `iterations`; проверка после валидации подписи, чтобы неподписанный
  NSEC3 не влиял на вердикт). Счётчики secure/insecure/bogus на дашборде (`QueryLog.dnssec_breakdown/0`).
- **Loop 7 — secure denial-of-existence** ✅. `prove_denial`: пустой ответ (NXDOMAIN/NODATA) с SOA →
  собираем цепочку до апекса (owner SOA), валидируем подписанные NSEC/NSEC3, доказываем отсутствие
  имени/типа. Поддержано: NSEC **NODATA** (owner == имя, тип отсутствует), NSEC **NXDOMAIN cover**
  (каноническое сравнение имён RFC 4034 §6.1), NSEC3 **NODATA** (match + тип отсутствует), а также
  NSEC «black lies» (Cloudflare/ietf отдают NXDOMAIN как NODATA). Доказано → `:secure`, иначе →
  `:insecure` — **никогда `:bogus`**, чтобы не SERVFAIL'ить легитимный NXDOMAIN неподдерживаемого
  стиля. Проверено живьём: cloudflare/ietf (black lies) и nlnetlabs.nl (классический NSEC) → secure;
  битая подпись доказательства → insecure.
  - *Остаётся (по желанию, не security):* NSEC3 closest-encloser proof для NXDOMAIN самого ответа
    (сейчас fallback в insecure), wildcard-доказательство, догон CNAME-цепочки до target.

## 9. Как проверить

```bash
make test          # весь набор; DNSSEC-тесты в test/eli_hole/dnssec/
make precommit     # format + credo + sobelow + полный тест
```

Ключевые проверки в тестах:
- key tag корневого KSK = 20326;
- DS-дайджест DNSKEY совпадает с хардкод-якорем;
- валидная RRSIG (RSA и ECDSA) проходит, мутация байта подписи/RRset — нет;
- неподдерживаемый алгоритм и мусорный пакет не роняют процесс, а возвращают ошибку/false.
